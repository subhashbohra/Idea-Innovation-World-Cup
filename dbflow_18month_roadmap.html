Phase 0 — foundation (weeks 1–8) · you, solo
The first step you answer with on day one
No new headcount. Leverage your existing Iceberg/Trino/Nessie/Airflow/Ollama stack. Produce a working demo in 8 weeks.
Weeks 1–2 · Data inventory and scoping
Identify which internal flow datasets (FX, trade finance, prime brokerage) are already in Iceberg tables. Audit data lineage, PII flags, and classification levels. This is not engineering — it is compliance-first discovery. Output: a one-pager of approved, shareable datasets and their schemas.
Weeks 3–4 · Feature store design
Design the flow signal schema on top of Iceberg. Each row = a flow event (asset class, direction, notional band, counterparty sector, timestamp). No raw client data. Aggregated at a level that passes data classification. Build the Airflow DAG that populates it daily from source Trino queries.
Weeks 5–6 · RAG layer over internal research
Use your existing Ollama/Gemma infrastructure. Chunk DB's internal research notes, macro commentary, and sector reports into pgvector. Build a LangChain retrieval chain that takes a natural language query, retrieves relevant flow signal rows + research context, and generates a structured answer. No fine-tuning yet — RAG first.
Weeks 7–8 · Live demo interface
React frontend, single-page. Left panel: NL query input. Right panel: LLM-generated insight card with sourced flow signals below it. Three canned demo scenarios: (1) "What is the FX positioning in EM currencies this week?", (2) "Show me trade finance flow trends in Asia Pacific", (3) "What are hedge funds doing in European credit?". This is the Innovation World Cup deliverable.
First question you will be asked: "Where is this data from and is it approved?" Answer: you audit classification in Week 1 before touching anything. You only use aggregated, non-attributable flow signals — same data DB already publishes in its quarterly flow reports, just queryable in real time.
Phase 1 — PoC to internal pilot (months 3–6) · you + 2 people
Prove it works with a real internal team
Bring in one FIC quant analyst (domain expert) and one ML engineer. Target: 3 internal DB business units using it daily.
Month 3 · First internal pilot — FIC desk
Partner with one FX or rates desk. Let traders query the system daily. Collect feedback on signal quality, query latency, and hallucination rate. Define accuracy metrics: does the LLM's answer match what a senior analyst would say? Track weekly.
Month 4 · LoRA fine-tuning pipeline
Your ML engineer runs LoRA fine-tuning on Gemma using the approved internal research corpus — not raw flow data, but commentary about flows. This tunes the model's financial language without exposing client positions. Infrastructure: GKE job, Hugging Face PEFT library, checkpoint storage on GCS.
Month 5 · Multi-asset expansion
Extend the feature store from FX-only to trade finance and correspondent banking flows. Each asset class needs its own Iceberg partition scheme and Airflow DAG. The quant analyst owns the signal logic; you own the pipeline. Output: 3 asset class modules queryable from the same interface.
Month 6 · Pilot review and architecture sign-off
Present pilot results to DB architecture board (you already have a relationship from the RCA approval). Metrics to show: query volume, answer accuracy rate, time-to-insight vs manual research, NPS from pilot traders. Get formal sign-off to proceed to external pilot.
You — platform architect + product lead FIC quant analyst — domain validation ML engineer — fine-tuning + inference
Phase 2 — external alpha (months 7–12) · you + 5 people
First paying client. Real revenue signal.
Add compliance, API, and client-facing layers. Target: 3 paying external clients (2 hedge funds, 1 corporate treasury) generating first ARR.
Month 7–8 · Compliance and data governance layer
This is the hardest non-technical step. Work with DB Legal and Data Governance to classify each flow signal type under MiFID II. Instrument data lineage end-to-end using your Nessie catalog — every signal row must be traceable to its source. Build consent framework: which data can be shared externally, under what license terms.
Month 9 · External API layer
Build a rate-limited REST + WebSocket API gateway on GKE. Multi-tenant auth (OAuth2 + API keys). Per-client namespace isolation. Rate limiting by tier (Tier 1 = 100 queries/day, Tier 2 = unlimited with SLA). Python SDK wrapping the REST API — this is what hedge fund quant teams will actually use. OpenAPI spec auto-generated from FastAPI.
Month 10–11 · Alpha client onboarding
DB's IB client coverage team introduces dbFlow to 5 hedge fund contacts. You (as product lead) do the technical demo. First 3 clients get 90-day free access in exchange for weekly feedback sessions. You are measuring: which query types generate the most value, what data is missing, what the willingness-to-pay signals look like.
Month 12 · First paid contracts
Convert 2–3 alpha clients to paid Tier 1 or Tier 2 subscriptions. $50k–$200k/yr each. This is the moment the idea becomes a business. Revenue is proof. Everything before this is cost.
You — product lead + external face FIC quant (x2) — signal quality ML engineer — inference + fine-tuning Compliance specialist — MiFID II, data licensing Backend engineer — API gateway, SDK
Phase 3 — productization and scale (months 13–18) · full team
From product to platform. From 3 clients to 30.
Dedicated team of 7+. Self-service onboarding. Automated billing. SLA-backed. €20M ARR trajectory.
Month 13–14 · Self-service client portal
React dashboard: API key management, query usage analytics, signal module selection, billing portal. Clients can onboard without a DB relationship manager. This removes the human bottleneck that limits scale.
Month 15 · Bespoke Tier 3 delivery
For SWF-level clients: LoRA fine-tune a custom model slice on the flow data most relevant to their strategy (e.g. EM currency flows for a Gulf SWF). Dedicated GKE namespace, dedicated model endpoint, 99.9% SLA. This is the $1–5M/yr tier.
Month 16–18 · Monetization flywheel
Each new client's usage data improves the model's understanding of what signals matter. Usage analytics feed back into the feature store — if 30 hedge funds all query EM FX at the same time, that pattern itself is a signal. Build the feedback loop that makes dbFlow smarter with scale, not just bigger.
You — VP Product / Chief Architect FIC quants (x2) ML engineers (x2) Compliance + legal Backend engineers (x2) Frontend + UX Client success manager
